# 🚀 csautobot 견적 생성 최적화 및 연동 완료 보고서

**작업 일시**: 2026년 7월 14일  
**주요 대상**: `csautobot` (Python 백엔드), `aiCsms` (Java 연동 파트)

---

## 1. 작업 개요 및 배경
사용자가 "AI 견적 생성" 버튼을 클릭했을 때 **CPU 점유율이 100%로 치솟고 응답이 심하게 지연(또는 502/500 에러 발생)되는 현상**이 접수되었습니다. 주요 원인은 클라우드 LLM(Gemini, OpenAI 등)의 API 할당량(Quota) 초과 시 적절히 오프라인 모드로 전환되어야 하나, 그 과정에서 무거운 임베딩 연산과 인코딩 충돌 등 연쇄적인 예외가 발생했기 때문입니다. 

이를 해결하기 위해 **로컬 모델(Ollama) 최적화, 예외 처리 강화, 윈도우 인코딩 호환성 패치**를 진행하였으며, 외부 연동 플랫폼인 `aiCsms`와의 호환성 검증까지 완료했습니다.

---

## 2. 발생 이슈 및 시행착오 (Trials & Errors)

### 🚨 이슈 1: RAG 검색 시 CPU 100% 급증 및 속도 저하
* **원인**: 문서 검색 후 정확도를 높이기 위해 실행되는 `rerank_by_embedding` 과정에서 로컬 Ollama를 사용할 경우, 실시간으로 수십 개의 문서를 다시 임베딩하느라 CPU 자원이 과도하게 소모되었습니다.
* **시행착오**: OpenAIEmbeddings 대신 OllamaEmbeddings로 롤백하는 과거 커밋(`39c6de8`)을 강제로 적용해 보았으나, 최근에 추가된 사용량 측정(`AiUsageInfo`) 모듈과 충돌하며 백엔드 앱이 크래시(ImportError)나는 문제가 발생했습니다.
* **해결**: 최신 코드를 유지하면서, **Ollama(`nomic-embed-text`)를 사용할 때는 무거운 재임베딩(Reranking) 로직을 안전하게 우회(Bypass)**하도록 `retrieval.py`를 수정하여 CPU 과부하를 원천 차단했습니다.

### 🚨 이슈 2: 오프라인 Fallback 중 500 Internal Server Error 발생
* **원인**: 클라우드 API(Groq, Gemini 등) 한도 초과 시 오프라인 규칙 엔진으로 전환되는데, 실패 사유를 콘솔에 출력(`print`)하는 과정에서 사용된 **긴 줄표 특수기호(`—`, em-dash)**가 윈도우 서버의 기본 인코딩(`cp949`)과 충돌하여 `UnicodeEncodeError`를 발생시켰습니다.
* **해결**: `ai_provider.py` 및 `quotation_service.py` 내의 해당 특수기호를 일반 하이픈(`-`) 및 화살표(`->`)로 모두 교체하여, 윈도우 콘솔 출력 시 서버가 크래시되는 현상을 수정했습니다.

### 🚨 이슈 3: GitHub Actions 한도 초과로 인한 배포 지연
* **원인**: GitHub Actions 무료 사용량 한도 도달로 인해 자동 CI/CD 배포 파이프라인이 중단되었습니다.
* **해결**: PowerShell과 SCP를 활용한 **원격 수동 배포 스크립트**를 직접 실행하여 핵심 수정 파일(`quotation_service.py`, `retrieval.py`, `ai_provider.py`)만 타겟 서버(`C:\deploy\csautobot`)로 안전하게 덮어쓰기 한 뒤 PM2 데몬을 재시작해 다운타임을 최소화했습니다.

## 3. 최종 개선 사항 및 결과 (Improvements)

**성능 향상 및 안정성 확보**
로컬 임베딩 최적화를 통해 견적 초안 생성 속도가 눈에 띄게 단축되었으며, CPU 100% 점유 현상이 사라졌습니다.

1. **안전한 오프라인 전환 (Robust Fallback)**: 
   * API Rate Limit 발생 시, 에러를 내고 멈추는 대신 **오프라인 진단 결과와 함께 정확한 실패 사유("한도 초과/rate-limit")를 화면에 반환**하도록 개선했습니다.
2. **인코딩 충돌 방지**: 
   * 윈도우 OS 기반의 구동 환경을 고려하여, 시스템 로그 출력에 사용되는 모든 문구의 Unicode 호환성을 검증했습니다.
3. **`aiCsms` 완벽 연동 검증**: 
   * Java Spring 기반인 `aiCsms`가 csautobot의 응답을 파싱할 때, 새롭게 추가된 `ai_usage` 필드로 인해 역직렬화(Deserialization) 에러가 발생하지 않도록 `@JsonIgnoreProperties(ignoreUnknown = true)` 구조가 잘 적용되어 있음을 확인했습니다.
   * 결과적으로 aiCsms 프론트엔드 대시보드 화면에서도 수정된 오프라인 견적 결과 및 사유가 100% 동일하게 정상 출력됩니다.

## 4. 향후 권장 사항
- **GitHub Actions Quota 점검**: 현재 CI/CD 자동화가 리밋에 걸려 있으므로, 추후 팀 저장소의 Actions 요금제 또는 한도를 점검해주시면 향후 유지보수가 더욱 원활해집니다.
- **로컬 LLM 도입 고려**: 외부 클라우드 API 의존도로 인한 한도 초과가 잦을 경우, 견적 진단용 경량 로컬 LLM(예: Qwen 2.5 3B 등)을 서버에 상주시키는 하이브리드 구성을 본격적으로 확장하는 것도 좋은 해결책이 될 수 있습니다.
